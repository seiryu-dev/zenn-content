## TL;DR

- ノイズ除去プラグイン（無料/有料）を同一音源・同一条件で数値比較し、そのあと**同条件で実際に聴き比べて答え合わせ**をした。
- SNR改善の1本だけで見ると NVIDIA Broadcast が圧勝に見える。しかし「消しすぎ・作りすぎ検知」のレーンは**高域603〜6310%生成**と警告しており、実際に聴くと**「ぷつぷつ」というクリップ歪みが確認できた**（TruePeak +3.9 dBTP）。数値の警告を耳が裏付けた形。
- 一方、数値では13ポイント差があった無料 Bertom と有料 RX の高域保持（99% vs 86%）は、**このトーク素材では聴いて差が分からなかった**。当初「無料が有料に勝つ」と書きかけたが、それは言い過ぎだった。正しくは「聴感上ほぼ同等」。**数値差＝可聴差ではない。**
- おまけに、評価指標そのものも一度壊れていた（偽陽性）。つまり今回、**耳が正しくて数値が間違っていた回**と、**数値が正しくて耳が裏付けた回**の両方が起きた。「耳が常に正しい」でも「数値が常に正しい」でもない——これがこの記事の結論。

## なぜこれをやったのか

いま、音声整音の受託を準備している。人の作品を預かって加工する以上、「自分の耳に良く聞こえたから」では根拠として弱い。ノイズは消えたが息づかいまで削れていた——そういう失敗を、納品したあとに気づくわけにはいかない。

もともと組込みソフトの開発をしていて、単体テストで品質を担保する文化に慣れている。だから音でも同じことをしたくなった。**選定を勘ではなく数値で詰め、その数値が本当に意味を持つのかを耳で確かめる。** それが今回やったことだ。

宅録音声のノイズ除去プラグインは無料から数万円まで幅広くあるが、「どれがどのくらい効くのか」「どこから消しすぎになるのか」を定量的に比べた情報は意外と少ない。多くの比較は耳の印象ベースで、条件も揃っていない。

そこで、**正解データ（クリーン音源）がある状態**を人工的に作り、同一条件で測ってみた。そして最後に、数値の結論を**自分の耳で答え合わせ**した。先に白状すると、数値だけで書いた最初の結論は、耳によって一部修正されることになる。

## 実験設定

### 正解データを持っている、という珍しい条件

- 素材：Pixabay の日本語音声（商用可・クレジット不要）。54秒・48kHz・モノラル。
- この**クリーン音源に人工的にノイズを付与**して「Before」を作った。
  - ピンクノイズ ＋ 電源ハム（50/100/150Hz）
  - Before の SNR は 31.7 dB（宅録相当を想定）
- つまり「ノイズを足す前の音」が手元にある。**処理結果を"本来の声"と直接比較できる**。実録音ではまず得られない条件で、これが評価設計の土台になる。

### 処理パイプライン

全プラグインに共通の下処理を通したうえで、VST3プラグインを1段だけ掛けた。条件差はプラグインとその強度設定のみ。

```
入力 → 共通下処理（自作 clean.py＝ffmpeg パイプライン）
        ハイパス → ハム除去 → ノイズ除去(afftdn) → loudnorm -18 LUFS
     → VST3 プラグイン 1段 → 評価
```

### VST3 を DAW なしで回す（pedalboard）

比較実験では「同じ設定で何十回も再実行できること」が重要なので、DAW ではなく Spotify の [pedalboard](https://github.com/spotify/pedalboard) で VST3 をヘッドレスにロードした。

```python
from pedalboard import load_plugin
from pedalboard.io import AudioFile

plugin = load_plugin(r"C:\Program Files\Common Files\VST3\Denoiser.vst3")
print(plugin.parameters)          # パラメータ名と現在値を確認できる
plugin.amount = 0.6               # スクリプトから強度を振れる＝グリッド実験が可能

with AudioFile("before_processed.wav") as f:
    audio = f.read(f.frames)
    sr = f.samplerate

out = plugin(audio, sr)
```

パラメータをコードから振れるので、「0.3 / 0.6 / 0.9 で総当たり」のような実験が数分で回る。

## 評価指標の設計

ここが本題の半分。測り方を間違えると、結論ごと間違える。

### 方針：レーンを2本以上持つ

守りたいものを先に決めた。ノイズ除去の目的は「ノイズを減らすこと」だが、納品品質として本当に守りたいのは **声の質感（息づかい・空気感＝高域、声の芯＝中高域）** だ。なので評価レーンは最低2本要る。

1. **ノイズ低減レーン**：SNR改善（dB）
2. **消しすぎ・作りすぎ検知レーン**：帯域エネルギー保持率
   - 6–16 kHz（高域＝息づかい・空気感）
   - 2–6 kHz（中高域＝声の芯）

### 音量に依存させない

処理によって全体音量が変わるので、そのまま比較すると音量差を「音質差」と誤認する。対策として：

- SNR は絶対値でなく **フレームRMSの p90/p10 比**で測る（音量スケールに不変）
- 帯域保持率は **発話フレームのみを抽出し、RMS正規化してから**測る

```python
import numpy as np

def frame_rms(x, sr, frame_ms=50):
    n = int(sr * frame_ms / 1000)
    frames = x[: len(x) // n * n].reshape(-1, n)
    return np.sqrt((frames ** 2).mean(axis=1))

def snr_proxy_db(x, sr):
    """発話(p90)と無音(p10)のRMS比。音量に不変。"""
    rms = frame_rms(x, sr)
    return 20 * np.log10(np.percentile(rms, 90) / np.percentile(rms, 10))

# SNR改善 = snr_proxy_db(after) - snr_proxy_db(before)
```

帯域保持率は、**真のクリーン音源の帯域エネルギーを100%** として、処理後がその何%かを見る。100%未満なら消しすぎ、100%を大きく超えるなら「無かった成分を作っている」疑い。

### 落とし穴：「処理後 ÷ 入力」では消しすぎを測れない（数値が間違っていた回）

最初、正解データを使わずに「処理後の高域 ÷ 入力の高域」で消しすぎを判定しようとして、盛大に間違えた。

**入力の高域はノイズが支配的なことが多い。** だからノイズを正しく消しただけで高域エネルギーは大きく減り、「消しすぎ」と誤判定される（偽陽性）。実際、この誤った指標は「高域25%＝消しすぎ」と警告を出したが、**耳で聴くと明らかに良くなっていた**。数値の方が壊れていた。

修正はこうした：

1. **入力の無音部から定常ノイズのスペクトルを推定**する
2. 入力スペクトルからそれを差し引き、「本来の声」のスペクトルを推定する
3. 処理後をその推定値と比較する

```python
# 概念コード（要点のみ）
S_in    = band_psd(x_in,    sr, speech_frames)   # 入力・発話部のPSD
S_noise = band_psd(x_in,    sr, silence_frames)  # 入力・無音部のPSD＝定常ノイズ推定
S_out   = band_psd(x_out,   sr, speech_frames)

S_voice_est = np.maximum(S_in - S_noise, eps)    # 「本来の声」の推定
retention = band_energy(S_out, 6000, 16000) / band_energy(S_voice_est, 6000, 16000)
```

今回は真のクリーン音源があるので答え合わせができる。修正後の推定値は **91%（真値85%）** とほぼ一致した。正解データなしの実録音でも、この方式ならおおむね正しく消しすぎを検知できる見込みが立った。

これが1回目の「数値と耳の食い違い」で、このときは**耳が正しく、指標を直した**。この話にはあとで対になる続きがある。

### %ではなくdBで併記する

「保持率70%」は数字として強烈だが、dB換算では **−1.6 dB** で、聴感上は軽微な部類に入る。%表記は差を誇張して読み手（自分含む）を誤らせるので、判断はdBで併記して行った。

## 数値の結果

同一音源・同一の下処理・プラグイン1段。保持率は真のクリーン音源＝100%基準。

|---|---|---:|---:|---:|

| Bertom Denoiser 0.3 | 無料 | +5.2 dB | 99% | 100% |
| **Bertom Denoiser 0.6** | 無料 | **+11.2 dB** | **99%** | 101% |

## 耳で答え合わせする

記事を公開する前に、数値の結論を聴いて確かめた。方法は：

- 各処理結果から**同じ18秒**を切り出す
- **すべて -18 LUFS に音量統一**する（音量差を音質差と誤認しないため。数値評価と同じ思想）
- 同一環境で聴き比べる

正直に書くと、**盲検ではない**。自分で処理したファイルを、どれがどれか知ったうえで聴いている。だからここから先の「聞こえた／聞こえなかった」は、厳密な官能評価ではなく一人の耳の所見として読んでほしい。それでも、結論は3つ動いた。

聴取環境は、DAC/ヘッドホンアンプに **RME ADI-2/4 PRO SE**、イヤホンに **qdc Emperor**（マルチBA IEM）。据え置きのリファレンス環境で、イヤホンで細部を追う前提の構成にした。

### 聴いて分かったこと

1. **NVIDIA Broadcast は「ぷつぷつ」言っている。** 数値が警告していたアーティファクトが、耳ではっきり確認できた。
2. **Bertom と RX に大きな差はなかった。** 数値では高域保持に13ポイント差（99% vs 86%）があるのに、このトーク素材では可聴差にならなかった。
3. **Waves Clarity Vx を足すと、ノイズが「全部消えた」と感じる。** 実際に"消えた"という印象を与えたのは、これが唯一だった。
4. **RX 11 Voice De-noise は、ノイズを消す「強さ」が足りない。** その代わり、イコライザーのように**帯域ごとにアプローチできる**。

この4つが、数値だけで書いた結論をどう変えたか。順に書く。

## 耳を通したあとの発見

### 発見1（この実験の白眉）：数値が「生成」と警告し、耳が「ぷつぷつ」と裏付けた

SNR改善だけ並べると、NVIDIA Broadcast の +16.8〜+28.8 dB は他を圧倒している。**評価レーンが1本しかなければ、実験は「NVIDIA の勝ち」で終わっていた。**

しかし消しすぎ・作りすぎ検知レーンは別のことを言っていた：

- 高域保持率が **603〜6310%**。100%を桁違いに超えている＝ノイズを消したのではなく、**元の声に存在しなかった高域成分を大量に生成している**（アーティファクト）
- 中高域（声の芯）は78%まで削れている（0.5時）
- TruePeak **+3.9 dBTP** で 0 dBTP を超えてクリップ

そして実際に聴くと、**「ぷつぷつ」というクリップ歪みが乗っていた**。原因もはっきりしている。TruePeak が 0 dBTP を超えた時点で**波形はすでに潰れている**ので、後段で音量を下げても歪みは残る。数値の警告 →「予測」、聴取での確認 →「検証」。この一本の線がつながったのは、**評価レーンを2本持っていたから**だ。SNR 1本なら、この歪んだ出力を勝者として納品候補にしていた。

公平のために書くと、NVIDIA Broadcast は**リアルタイム配信用に設計されたツール**であり、遅延ゼロ近くで会議やゲーム配信の環境音を消すという本来の用途では優れた選択肢だ。納品やマスタリング用途に最適化されていないだけで、悪い製品という話ではない。**用途外のものを単一指標で測ると勝者に見えてしまう、という測る側の問題**である。

### 発見2：高域13ポイント差は、聞こえなかった —「無料が勝つ」の撤回

最初のドラフトでは「無料の Bertom 0.6 が有料 RX に勝った」と書いていた。**これは言い過ぎだったので撤回する。**

数値上の事実は変わらない。Bertom 0.6 は +11.2 dB のノイズ低減で高域99%を保持し、RX 11 はほぼ同等のノイズ低減（0.4で+12.5 dB）の時点で高域が86%まで落ちる。13ポイントの差だ。しかし**同じ18秒・同一ラウドネスで聴き比べたら、大きな差は感じられなかった。**

正しい結論はこうなる：**無料の Bertom は、有料の RX と聴感上ほぼ同等**（数値上は高域をより多く保つが、このトーク素材ではその差は可聴差にならなかった）。

ここには指標設計と同じくらい重要な教訓がある。**指標は「差がある」とは言えても、「その差が聞こえる」とは言えない。** 数値差と可聴差は別物で、可聴性を主張したければ ABX テストのような別の実験で確かめる必要がある。今回の聴取は盲検ですらないので、「差がなかった」の側もあまり強くは言えない——だからこそ「ほぼ同等」までしか書かない。

なお、息づかいや空気感そのものが商品になる ASMR・ささやき系の素材では、高域13ポイントの差が可聴差として効いてくる**可能性**はある。ただし未検証なので、これは可能性以上のことではない。

念のため書くと、RX はスペクトル修復など比較外の機能を多く持つ統合ツールで、この1項目で製品の優劣が決まるわけではない。

### 発見2b：RX は「強く消す道具」ではなく「帯域ごとに削る道具」だった

聴いていて、もうひとつはっきりしたことがある。**RX 11 Voice De-noise は、ノイズを消す「強さ」そのものは足りない。** その代わり、**イコライザーのように帯域ごとにアプローチできる**。

これはパラメータを見ると腑に落ちる。RX Voice De-noise が公開しているパラメータは `reduction`（全体の削減量）だけではなく、`threshold_1` 〜 `threshold_6` という**6本の帯域別しきい値**を持っている。つまり設計思想が「全体を強力に消す」ではなく「**帯域ごとに外科的に削る**」側に寄っている（実際、フィルタタイプの既定値も "Surgical" だった）。

そう考えると、数値の挙動とも一致する。RX は強度を上げても高域保持が86%前後の水平線に張り付き、それ以上は下がらない。強く消しにいくのではなく、帯域ごとに一定量を削って止まる——**その削り方が高域に一定の代償として出ている**、という読み方ができる。

「消す強さ」で選ぶ道具ではなく、「どの帯域をどれだけ削るか」を制御する道具。用途に対して道具を取り違えると、「効かない」という誤った結論になる。**数値の形（水平に張り付く）と、耳の印象（強さが足りない）と、パラメータの設計（6帯域・Surgical）が、同じことを別々の角度から言っていた**のは面白かった。

### 発見3：強くすれば良いわけではない（純損ゾーンは数値でしか見えない）

Bertom 0.9 は 0.6 より SNR改善が**むしろ悪化**（+10.9 dB）したうえ、高域が99%→85%に落ちた。ノイズ低減も音質も両方悪くなる、つまり **純損**。

これは表を眺めているだけだと見落としやすいし、耳だけで強度を追い込んでいても気づきにくい。強度を横軸に取った「かけすぎマップ」を描いて初めて、「このプラグインの美味しいゾーンはここまで」という崖の位置が見える。数値レーンが耳より先に働く場面の一例だ。

### 発見4：「消えた」と感じさせたのは NN 系 — ただし"保存"ではなく"再構成"

聴き比べでもうひとつはっきりしたのは、ノイズが「全部消えた」という印象を与えたのは **Waves Clarity Vx（ニューラルネット系）を足した構成が唯一**だったことだ。数値でも SNR改善 +28.3 dB で最大。

ただし帯域保持を見ると、中高域（2–6k）70%・高域（6–16k）200%。つまり**元の声をそのまま保存しているのではなく、帯域バランスを作り変えている**——"保存"というより"再構成"に寄った処理だ。

これは良し悪しではなく使い分けの話になる。トークや朗読のように「明瞭に聞こえること」が価値の素材なら、再構成されても聴感が良ければそれでいい。逆に ASMR のように**元の質感そのものが商品**の場合、帯域バランスを作り変える処理は商品の中身を変えてしまうリスクがある。そして、この使い分けの判断こそ、耳の印象（「消えた」）と指標（保持率200%/70%）を**並べて見ながら**下すべきものだと思う。耳だけなら「消えたから最高」で終わり、数値だけなら「⚠付きだから除外」で終わっていた。

## 教訓：数値と耳、どちらか一方を信じると間違える

今回の実験から持ち帰ったことを一般化すると、こうなる。

1. **単一指標は最適化のハックに弱い。** SNRだけ見れば「高域を生成で埋め、クリップした」出力が最高スコアを叩き出す。守りたい性質ごとにレーンを分ける。
2. **指標の分母（基準）を疑う。** 「処理後÷入力」は一見自然だが、入力がノイズ混じりである以上、正しい処理を「消しすぎ」と誤判定する。基準は「本来あるべき姿」の推定に置く。
3. **数値差は可聴差ではない。** 指標は「差がある」ことしか言えない。その差が意味を持つか（聞こえるか）は、別の検証がいる。13ポイント差が聞こえなかった今回がまさにそれ。
4. **表示スケールで判断が変わる。** %は差を誇張する。dBなど、対象領域で意味のあるスケールで併記する。
5. **数値と耳が食い違ったら、まず指標を疑う。ただし指標が正しいこともある。** 今回は両方起きた。①「処理後÷入力」の偽陽性では耳が正しく、指標を直した。②NVIDIAの生成警告では数値が正しく、耳が「ぷつぷつ」で裏付けた。この2つが両方あることが重要で、**「耳が常に正しい」でも「数値が常に正しい」でもない**。数値は耳で気づけない破壊（あるいは崖の位置）を捕まえ、耳は数値が意味を持つかを確かめる。どちらかを削ると、今回の間違いのどちらかをそのまま公開していた。

これは音声に限った話ではないと思っている。LLMの出力評価でも、ベンチマーク1本で選ぶと「そのベンチに過適合した出力」を勝者と誤認する構造は同じだ。そしてベンチのスコア差が実利用での体感差になるとは限らない、という点まで含めて同じだと思う。

本業の車載組込みでも、似たことは起きる。**単体テストが全部パスしても、それは「テスト設計が正しかった」ことを意味しない。** カバレッジを満たしていても、見るべき観点が抜けていれば、欠陥は通ったまま残る。テストが緑になったかどうかより、「**このテストは何を保証しているのか**」を疑うほうが本質だと思っている。

今回の SNR も同じだった。指標を満たすことと、守りたいものが守られていることは、別の話だ。**測って安心するのではなく、その測り方が何を見落としうるかを先に考える。** 組込みでもLLMでも音声でも、そこは変わらないと思う。

## 再現方法

- 使用ツール：Python + [pedalboard](https://github.com/spotify/pedalboard)、ffmpeg、numpy/scipy
- **無料プラグイン（Bertom Denoiser）だけで主要な結果は再現できる。** NVIDIA Broadcast は RTX GPU が必要。
- 素材は Pixabay 等の商用可・クレジット不要の音声にノイズを合成すれば、誰でも「正解データあり」の条件を作れる。
- 聴き比べをする場合は、**同じ区間を切り出し、ラウドネスを揃える**こと（今回は18秒・-18 LUFS）。音量が揃っていない聴き比べは、ほぼ音量の比較になる。
- コード一式はここに置いた：**https://github.com/seiryu-dev/seion-eval**
  - `clean.py`（ffmpeg 下処理パイプライン）／`preflight.py`（素材診断）／`finish.py`（pedalboard で VST3 をヘッドレス適用）／`plugin_probe.py`（インストール済み VST3 の自動検出とパラメータ列挙）
  - `eval/`（評価ハーネス）／`experiments/`（かけすぎマップの再現スクリプト）
  - `tuner/`（GUI。**プラグインのネイティブUIを複数同時に開いたまま、再生しながらリアルタイムに詰められる**。プラグインは任意で、1つも無くても ffmpeg だけで動く）
  - `make_sample.py`（商用可の音声にノイズを合成して「Before」と「正解」のペアを作る）

## 今後やりたいこと

- **実録音での検証**。今回は人工ノイズなので「正解データがある」代わりに、現実のノイズ（空調・電源・部屋鳴り）とは性質が違う。
- **ABX による可聴差の検証**。今回いちばん歯切れが悪かったのがここで、「13ポイント差が聞こえなかった」は盲検で確かめないと本当は言い切れない。
- **ASMR・ささやき素材での再測定**。高域が商品そのものになる素材では、今回「聞こえなかった」差が効いてくる可能性がある。
