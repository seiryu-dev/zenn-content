## この記事について

自宅のPCで、自作のLLMゲートウェイを常時稼働させています。趣味のデモではなく、日々の仕事（副業の受託作業・調査・実装）が実際にその上を通っています。止まると自分の作業が止まる、という意味で「本番」です。

この記事では、**コードもconfigも出しません**。APIキー・受託先のデータ・個人環境のパスが混ざっているからです。代わりに、

- どういう構成にしたか（構成図）
- なぜその設計にしたか（判断の理由）
- **実際に起きた障害と、そのとき何を誤診しかけたか**

の3つで書きます。個人的には、LLM基盤の学びはこの3つ目に一番濃く詰まっていました。

書いているのは、車載組込みのSES出身、現役エンジニアです。以前、**LLM出力の評価設計**について別の記事を書きました（[seion-eval](https://github.com/seiryu-dev/seion-eval) と同シリーズ）。今回はその対になる「基盤側」の話です。「出力をどう測るか」と「呼び出しをどう安全に流すか」は、結局セットでないと運用が成立しないと感じています。

### なぜ自分でゲートウェイを作ったのか

きっかけは、**モデルには得意・不得意がある**と気づいたことでした。たとえば普段の相棒は Claude ですが、**画像生成は得意ではありません**。なら画像は別のモデルに任せればいい。そう考えると、「1つのモデルに全部やらせる」前提そのものが間違っていて、**用途ごとに適したモデルへ振り分けたい**——それを毎回コードに手書きするのではなく、**呼び出し口を1つにまとめて、その裏で振り分ける**ものが欲しくなりました。

もうひとつ、これが自分にとっては大きい理由でした。**1つのモデルだけを使っていると、思考が偏る。** 同じモデルは同じ癖で答えるので、間違いに気づけない。だから **AI同士に辛口レビューをさせたり、壁打ちをさせたりして、意図的に別ベンダーの視点をぶつける**ようにしました。これをやるには、複数のプロバイダを**同じ口から、同じ形式で**呼べる必要があります。ゲートウェイはそのための土台です。

つまり、自分にとってのゲートウェイは「コストを絞る仕組み」である以前に、**適材適所で振り分け、視点を偏らせないための仕組み**でした。安全（顧客データの境界）とコスト（暴走の防止）は、そのうえで**必ず要るもの**として乗ってきた、という順番です。

## 1. 全体構成

まず全体像です。

```
[頭: オーケストレータ] --タスクJSON--> inbox/ --監視--> [Runner]
                                                          |
                                                          | OpenAI互換API
                                                          v
                                                   LiteLLM (proxy :4000)
                                                     |
                                                     ├─ direct API : OpenAI / Anthropic / Google
                                                     │     ← client-*  (顧客データOK)
                                                     │
                                                     └─ breadth    : OpenRouter
                                                           ← misc-*   (顧客データ禁止)

[頭] <--結果JSON-- outbox/ <----------------------------- [Runner]

           ↑ PII関所(匿名化) / 死活監視 / コスト管理 が全経路に噛む
```

役割は素朴に分かれています。

- **頭（オーケストレータ）**: 何をやるかを決め、タスクをJSONで書き出す。
- **inbox / outbox**: ファイルベースのキュー。頭と手の**契約（スキーマ）**はここに集約されます。
- **Runner**: inboxを監視し、拾ったタスクを実行して、結果をoutboxに返す。
- **LiteLLM proxy**: 各プロバイダの差異を吸収する、OpenAI互換の一枚。

そして、この経路の**全部**に、PII関所・死活監視・コスト管理が噛んでいます。「特定の呼び出しだけ安全」ではなく、「この配管を通る限り安全」にしたかったからです。

### 1タスク1実行・自律連鎖なし

設計上、意識的に**やらなかった**ことがあります。**Runnerに自律連鎖をさせない**。

1タスクを受け取ったら、1回実行して、結果を返して終わりです。「結果を見て自分で次のタスクを作る」ことはしません。次に何をするかを決めるのは常に頭側で、しかも人間が見られるファイル（inbox/outbox）を必ず経由します。

個人開発でエージェントを組むと、つい「勝手に走り続ける」方向に憧れます。ただ、自分の財布と自分のPCの上で動かしている以上、暴走のコストを払うのも自分です。**進捗が見える形で止まる**ことのほうを優先しました。

## 2. なぜゲートウェイを挟むのか

SDKを直接叩けば動くのに、わざわざ1レイヤ挟む理由は4つありました。

**(1) 用途別ルーティング**
「安いモデルで大量に回したい」「長文を投げたい」「調査だけは最上位モデルで」——用途はタスクごとに違います。呼び出し側のコードにモデル名を直書きすると、モデルが変わるたびに全部書き換えることになります。**呼び出し側は「用途」だけ言い、モデルの実体はゲートウェイが決める**。これが一番効きました。

**(2) 顧客データの境界**
受託作業を通すので、「このデータは外に出してはいけない」が実在します。人間の注意力でこれを守るのは無理です。**機械的に守る仕組み**が要る。次章で詳しく書きます。

**(3) コスト暴走の防止**
上限は一箇所に置きたい。呼び出し側それぞれが上限を持っていたら、誰も全体を把握できません。

**(4) 可観測性**
落ちたときに「どこが落ちたか」がわからないのが一番つらい。1本の配管にしておけば、監視も1箇所で済みます。

## 3. 名前空間を安全装置にする（設計の白眉）

ここが、この基盤で一番自分が気に入っている部分です。

ルーティングは**用途別エイリアス**でやっています。現在13ルート。呼び出し側が指定するのはモデル名ではなく、こういう名前です。

| エイリアス | 用途 | 解決先 |
|---|---|---|
| `client-smart` | 高品質が要る作業 | direct API |
| `client-fast` | 安価・速い | direct API |
| `client-long` | 長文コンテキスト | direct API |
| `client-local` | 完全ローカル | Ollama |
| `misc-cheap` | 安価・大量 | OpenRouter |
| `misc-smart` | 高品質（社外データなし） | OpenRouter 等 |
| `misc-research-*` | 調査・クロスチェック | 各社の最上位モデル |
| `image-default` | 画像生成 | 画像モデル |

ポイントは名前の**接頭辞**です。

### `client-*` は direct API か local にしか解決しない

`client-*` のルートは、**OpenAIやAnthropicやGoogleの直APIか、ローカルのOllamaにしか結線されていません**。OpenRouterのような、多数のモデルを束ねる「breadth系」の経路には**そもそも解決先が存在しない**。

`misc-*` は breadth系に流れてよい。ただし、**`misc-*` には顧客データを入れない**、という運用ルールとセットです。

つまりこうなります。

> **「どのルートを選ぶか」が、「顧客データが外へ出ないか」を機械的に決める。**

これは、レビューやチェックリストで守る類の安全ではありません。**呼び出し側が `client-` と書いた時点で、breadth系に流れる経路が物理的に存在しない**。設定の書き間違いで顧客データがOpenRouterへ飛ぶ、という事故の**形**がなくなります。

逆に言うと、この設計を裏切れるとしたら「`misc-*` に顧客データを手で入れる」ときだけです。守るべき人間の判断が1点に絞られている。これが狙いでした。

組込みをやっていた頃、「安全は運用手順ではなく回路で担保しろ」と何度も言われました。あれと同じことを、LLM呼び出しでやりたかったのだと思います。

### PII関所

もう一枚、匿名化の関所を置いています。機密にあたる文字列を**決定的に**伏字化し、結果が返ってきたら復元する。LLMは使いません（LLMに「秘密を隠して」と頼むのは、隠す判断自体をブラックボックスに委ねることになるので）。**round-trip（伏字化→復元で元に戻る）を検証済み**の、素直な決定的変換です。

`client-*` は既定で匿名化ONにしています。「必要なときにONにする」ではなく、「既定でONで、外すときに理由が要る」向きにしました。

## 4. コスト上限と死活監視

### コスト

二段構えです。

1. **プロバイダ側の利用上限**（そもそも突き抜けられないようにする）
2. **タスク単位の予算チェック**——送信前に概算トークンを見積もり、そのタスクの予算を超えるなら**実行を拒否する**

2つ目が効きます。「使いすぎたら止める」だと、気づいたときには使い終わっています。**投げる前に断る**ようにしておくと、事故の上限がタスク1件分に収まる。

### 死活監視

ゲートウェイとRunnerを、HTTPとプロセスの両方で定期的に叩いています。応答が古く（stale）なったら自動で再起動し、状態を `health_status.json` に書き出す。

「なぜファイル？」と思われそうですが、**落ちているときに読めるものが欲しかった**からです。監視の状態を凝った仕組みに載せると、その仕組みごと落ちたときに何も分からなくなります。テキストファイルなら、最悪エディタで開けます。

## 5. 運用でハマった6つ

ここからが本題です。設計はきれいに書けますが、実際に自分の時間を溶かしたのはここでした。**症状 → 誤診しかけた道 → 本当の原因 → 教訓**の形で並べます。

### 5-1. モデルIDを推測して404を踏んだ

**症状**: 新しいモデルのルートを足した。ゲートウェイの起動ログには、そのルートがちゃんと登録されている。なのにリクエストを投げると **404**。

**誤診しかけた道**: 「ルーティング設定のミスだろう」。ゲートウェイ側のyamlを何度も見直しました。ログには出ているのに、なぜ。

**本当の原因**: **上流のGoogleが404を返していた**。つまり「そんなモデルは無い」。モデルIDを**推測で**書いていたのです（`gemini-3.1-pro` / `gemini-3.5-pro` のように）。

**ListModels API に聞いて実測**したところ、正しくは **`gemini-3.1-pro-preview`** でした。`gemini-3.1-pro` というIDは**存在しない**。さらに、**3.5系にPROは存在せずFlashのみ**でした。命名は自分が想像したルールに従ってくれていませんでした。

**教訓**: **モデルIDは推測しない。プロバイダのモデル一覧APIに聞いて確定する。** 「たぶんこういう名前だろう」は、404という形で必ず返ってきます。しかもその404は、自分の設定ミスに見える顔をしてやってきます。

### 5-2. 「自動リロード」を信じていたら、古い設定のまま動いていた

**症状**: 5-1のモデルIDを正しく直した。それでもまだ404。

**誤診しかけた道**: 「じゃあIDもまだ違うのか？」——実測して確定させたIDを、もう一度疑い始めました。

**本当の原因**: **設定が反映されていなかった**。ホットリロードを意図した ConfigWatcher を実装してあり、自分はそれが効いていると信じていました。実際には、変更が反映されていなかった。**サービスを明示的に再起動して、初めて新ルートが通りました**。

**教訓**: **ホットリロードは「入れた」ではなく「効いている」を毎回確認する。** 一番たちが悪いのは、「自分が作った便利機能を、自分が無条件に信じてしまう」ことでした。実装した本人だからこそ、動いていることにしてしまう。デバッグ中に「今この瞬間、プロセスが読んでいる設定は本当に最新か」を確かめる手順を、最初の1手に入れるべきでした。

### 5-3. 429の正体はレート制限ではなく「残高切れ」だった

**症状**: Gemini直APIが **429** を返し続ける。

**誤診しかけた道**: 429 = Too Many Requests。素直に読めばレート制限です。「しばらく待てば直る」「リトライ間隔を延ばそう」——そう判断しかけました。

**本当の原因**: **エラー本文にこう書いてありました。**

```
RESOURCE_EXHAUSTED: Your prepayment credits are depleted
```

**プリペイド残高切れ**でした。待っても、永久に直りません。

**教訓**: **ステータスコードで判断しない。エラー本文まで読む。** HTTPステータスは「粗い分類」であって「原因」ではありません。429は「何かの枠を使い切った」であって、「時間を空ければ回復する枠」とは限らない。ステータスコードだけ見て自動リトライを組んでいたら、静かに永久ループしていたはずです。

### 5-4. 常駐サービスを SYSTEM で動かしていたせいで、認証に届かなかった

これが一番「基盤っぽい」失敗でした。

**症状**: あるCLIツールが、対話シェルからは正常に動く。ところが**Runner経由（サービス経由）だと必ず401**。認証情報は間違っていないはずなのに。

**誤診しかけた道**: 「トークンの期限切れか」「環境変数の渡し忘れか」。何度も再ログインし、環境変数を渡し直しました。直りません。

**本当の原因**: **実行ユーザーでした。**

RunnerはNSSMでWindowsサービス化してあり、**LocalSystem（SYSTEM）**として動いていました。実測するとこうです。

```
os.getlogin()  ->  SYSTEM
USERPROFILE    ->  C:\WINDOWS\system32\config\systemprofile
```

SYSTEMで動いているプロセスは、**ログインユーザーのWindowsキーリング（資格情報マネージャ）に保存された認証情報を読めません**。そのCLIツールは、認証をキーリングに置くタイプでした。だから対話シェル（＝自分のユーザーとして動く）では通り、サービス経由（＝SYSTEM）では通らなかった。401は、まったく正しい応答だったわけです。

一方で、**認証をファイルに置くタイプのツール**（例：`C:\path\to\.codex\auth.json` のような）は、**環境変数でパスを明示すればSYSTEMからでも読めました**。同じ「認証」でも、置き場所によってサービス化への耐性がまったく違う。

**教訓**: **常駐プロセスの「実行ユーザー」は設計事項である。**

ユーザーコンテキストに依存するもの——**キーリング / ユーザープロファイル / `%LOCALAPPDATA%` / ユーザー単位のPATH**——は、**サービス化した瞬間に届かなくなります**。ローカルで動くものが本番で動かない理由の多くは、コードではなく「誰として動いているか」でした。

サービス化を検討する時点で、依存しているツールが認証をどこに置くタイプなのかを確認しておくと、この401を回避できます。逆にこれを知らないと、認証情報そのものを延々疑い続けることになります（なりました）。

### 5-5. モデルの自己申告は当てにならない

**症状**: `--model` フラグでモデルを指定しているつもりだったが、**本当にそのモデルが呼ばれているのか確信が持てない**。

**誤診しかけた道**: モデル自身に「あなたは何のモデルですか」と聞きました。返ってきた名前で検証しようとした。——**当てになりませんでした**。自己申告は、システムプロンプトや学習時点の情報に引きずられます。これを証拠に採用するのは危険です。

**本当にやったこと**: **存在しないモデル名を渡してみる**。サーバー側が **400** で弾けば、`--model` の指定は確かにサーバーまで届いている（＝フラグは効いている）という**証明**になります。何食わぬ顔で動いたなら、フラグは無視されているということです。

**教訓**: **モデルの同一性は自己申告でなく、APIの振る舞いで確かめる。** LLMに「自分について」聞くのは、被験者に自己申告させる検査です。それより、**外側の観測可能な振る舞いで反証する**ほうが強い。「正しい入力で正しく動く」ことより、「不正な入力が正しく拒否される」ことのほうが、経路が生きている証拠になります。

### 5-6.（おまけの学び）ログが正しくても、経路は生きていない

上の6つを並べて気づいたことがあります。**5-1も5-2も、「自分側のログは正常だった」**のです。ルートは登録されている。ConfigWatcherは動いている（ように見える）。

自分の観測点が「正常」を出していても、それは**自分の観測点までが正常**というだけでした。エンドツーエンドで、実際に上流が何を返しているかを見に行くまで、原因には辿り着けませんでした。可観測性を「自分のログ」だけで組むと、こういう盲点ができます。

## 6. サブスク枠を使う経路と、フォールバック

### API課金を使わない経路

ある提供元のCLIは、**アカウントでサインインすると、APIの従量課金ではなくサブスクリプションに含まれる利用枠を使います**。これをスクリプトから**非対話（headless）**で叩けるようにしました。

結果として、**調査・クロスチェック用のモデル呼び出しをAPI課金ゼロにできました**。個人開発でLLMを日常的に回すとき、「調べもの」と「セカンドオピニオン」の呼び出しは地味に多く、しかも品質を落としたくない部分です。ここをサブスク枠に逃がせたのは大きかった。

ただし注意点があります。**レート枠は本体（対話での利用）と共有**です。スクリプトから乱発すると、自分の普段の作業のほうが詰まります。だから**乱発しない設計**にしてあります——連鎖させない、必要なときだけ、というのは §1 の「1タスク1実行」とも噛み合っています。

（ここは特定製品でできる話ですが、宣伝したいわけではないので淡々と。要は「課金経路は1本とは限らない」という話です。）

### フォールバック

同じ用途に対して、**複数プロバイダのルートを持たせています**。あるベンダーの枠が尽きても、別ベンダーのルートで同等の作業が回る。

5-3の「残高切れの429」を踏んだあと、これの価値が身に沁みました。**単一プロバイダに依存していると、残高が尽きた瞬間に基盤ごと止まります**。用途別エイリアスにしておいた効果がここで出ます——呼び出し側は用途しか言っていないので、**ゲートウェイ側で解決先を差し替えるだけで復旧できる**。呼び出し側にモデル名を直書きしていたら、全箇所を書き換えることになっていました。

§2 の「呼び出し側は用途だけ言う」は、可読性のためではなく、**この日のため**の設計だったと後から分かりました。

## 7. まとめ：難しさはモデルではなく「境界・観測・失敗の読み方」にある

この基盤を日々使い続けてきて、一番はっきりしたのはこれです。

**LLM基盤の難しさは、モデル選定にはほとんどありませんでした。**

時間を溶かしたのは全部、こういうところでした。

- **境界**: 顧客データがどこまで行っていいのか。それを人間の注意力ではなく**名前空間**で機械的に決める。
- **観測**: 自分のログが「正常」でも、経路が生きているとは限らない。**上流が実際に何を返したか**を見に行く。
- **失敗の読み方**: 404は自分の設定ミスの顔をしてやってくる。429はレート制限の顔をしてやってくる。401は認証情報の顔をしてやってくる。**どれも顔で判断すると外します。** 本文まで読む、実行ユーザーまで疑う、存在しない入力で反証する。

これはたぶん、LLM特有の話ではありません。組込みでも、分散システムでも、同じことを言われてきました。ただ、LLMは「それっぽく動いてしまう」ぶん、**間違ったまま動いている状態に気づきにくい**。だからこそ、境界と観測を先に置く価値が高い、という実感を持っています。

前作で書いた**評価設計**（出力をどう測るか）と、今回の**基盤**（呼び出しをどう安全に流すか）は、自分の中では完全に対です。測れないものは運用できないし、安全に流せないものは測る以前の問題になる。

### 運用期間について

正直に書くと、**この基盤を動かし始めてまだ1ヶ月ほど**です。「1年運用しました」という話ではありません。

ただ、ここに書いた6つは**その1ヶ月で実際に踏んだもの**です。長く運用したから出てきた話ではなく、**複数プロバイダを1つの口にまとめて、常時稼働のサービスとして動かした瞬間から出てくる**類の問題でした。おそらく、同じ構成を組む人は同じ順番で踏みます。

### 今後やりたいこと

- **失敗の読み方を自動化する。** 今回いちばん時間を溶かしたのは「404 / 429 / 401 の本当の意味を人間が読み解く」ところでした。ステータスコードだけでなく**エラー本文まで見て分類し、人が読まなくても『残高切れ』『モデルID違い』『認証がサービスから見えていない』を判別して通知する**ところまで持っていきたい。
- **観測性をトレースまで上げる。** 今は死活監視と `health_status.json` 止まりです。OpenTelemetry でモデル呼び出し・リトライ・失敗原因・レイテンシ・コストを**トレースとして残す**と、上の自動分類とも噛み合います。
- **常駐サービスの実行ユーザーを設計し直す。** 5-4 は根本的にはここが原因なので、対症療法ではなく「このプロセスは誰として動くべきか」を決め直すのが筋です。
- **RAG のルートを足して、その品質を eval で測る。** 前作で書いた「守りたいものを決めて、それを測るレーンを持つ」という話は、そのまま LLM の出力評価にも当てはまります。**基盤（安全に流す）と評価（正しく測る）が繋がるのはここ**だと思っています。
